## <a name="introduction"></a>소개

이전 자습서에서 설명 했 듯이 ASP.NET 응용 프로그램 개발 환경에서 관련 콘텐츠를 복사 하는 프로덕션 환경에 포함 됩니다. 배포 일회성 이벤트를 하지만 대신 버그 때 또는 소프트웨어의 새 버전이 출시 될 때마다 발생 하는 것 되었거나 보안 문제를 파악 하 고 해결 합니다. ASP.NET 페이지를 복사 하는 경우 이미지, JavaScript 파일 및 이러한 파일을 사용 하 여 직접 작업을 처리할 필요가 없습니다 프로덕션 환경에 이러한 다른 파일 이후로 변경 되었습니다 마지막 배포 합니다. 복사할 수 있습니다 맹목적으로 파일을 프로덕션에서 기존 콘텐츠를 덮어씁니다. 그러나 이러한 단순성 데이터베이스를 배포 하는 확장 되지 않습니다.

처음으로 데이터 기반 응용 프로그램을 배포 하는 경우 프로덕션 환경으로 개발 환경에서 데이터베이스를 맹목적으로 복사할 수 있습니다. 하지만 수행 하는 과제를 안겨 후속 배포에 복사 됩니다 데이터를 덮어씁니다 프로덕션 데이터베이스에 입력 합니다. 대신 데이터베이스를 배포 하는 것은 *변경 내용을* 프로덕션 데이터베이스로 마지막 배포 이후 개발 데이터베이스에 대 한 합니다. 이 자습서는 이러한 문제를 검사 하 고 chronicling 및 마지막 배포 이후 데이터베이스에 대 한 변경 내용을 적용에 도움이 되는 다양 한 전략을 제공 합니다.

## <a name="the-challenges-of-deploying-a-database"></a>데이터베이스를 배포 하는 문제

데이터 기반 응용 프로그램을 처음으로 배포 전에 데이터베이스가 하나만, 처음으로 데이터 기반 응용 프로그램을 배포 하는 경우 있습니다 무조건 이유는 개발 환경에서 데이터베이스에 데이터베이스를 복사 하는 namely를 프로덕션 환경으로 개발 환경입니다. 하지만 데이터베이스의 복사본을 두 가지 응용 프로그램을 배포한 후: 개발 및 프로덕션 환경에서 하나 하나입니다.

배포 간에 개발 및 프로덕션 데이터베이스에 동기화 될 수 있습니다. 프로덕션 데이터베이스의 스키마를 변경 되지 않은 동안 새 기능이 추가 됨에 따라 개발 데이터베이스의 스키마를 변경할 수 있습니다. 추가 하거나 열, 테이블, 뷰 또는 저장된 프로시저를 제거 될 수 있습니다. 개발 데이터베이스에 추가 되는 중요 한 데이터도 있을 수 있습니다. 사용자가 편집 가능한 되지 않은 하드 코드 된, 응용 프로그램별 데이터로 채워진 조회 테이블을 포함 하는 대부분의 데이터 기반 응용 프로그램입니다. 예를 들어 경매 웹 사이트를 auctioned 중인 항목의 상태를 설명 하는 옵션을 사용 하 여 드롭다운 목록의 있을: 새로 만들기와 같은 새, 정상, 및 Fair 합니다. 드롭다운 목록에서 직접 이러한 옵션을 하드 코딩 하는 대신 데이터베이스 테이블에 배치 하는 것이 좋습니다. 를 개발 하는 동안 저하 라는 새 조건이 추가 되 면 테이블에 다음 응용 프로그램을 배포 하는 경우이 동일한 레코드를 프로덕션 데이터베이스에서 조회 테이블에 추가 해야 합니다.

이상적으로 배포 데이터베이스는 수행할 개발에서 프로덕션 데이터베이스를 복사 합니다. 하지만 응용 프로그램을 배포 하 고 개발을 다시 시작 후 프로덕션 데이터베이스는 실제 사용자의 실제 데이터로 채워진 되 염두에서에 둡니다. 따라서 단순히 다음 배포 시 프로덕션 환경으로 개발에서 데이터베이스를 복사 하는 경우 프로덕션 데이터베이스를 덮어쓸 하는 해당 기존 데이터가 손실 됩니다. Net 결과 마지막 배포 이후 개발 데이터베이스에 대 한 변경 내용을 적용 하는 좁혀진 데이터베이스를 배포 합니다.

마지막 배포 이후의 스키마와 데이터의 변경 내용을 적용 해야 데이터베이스를 배포, 때문에 변경 내용의 기록을 유지 관리 (이거나 배포 시 결정) 해당 변경 내용을 프로덕션에 적용할 수 있도록 합니다. 관리 및 데이터 모델에 변경 내용을 적용 하기 위한 기술의 여러 가지가 있습니다.

### <a name="defining-the-baseline"></a>기준 정의

응용 프로그램 s 데이터베이스 변경 내용을 유지 하기 위해 일부 시작 상태, 변경 내용에 적용 되는 초기 계획 해야 합니다. 한편 시작 상태는 없는 테이블, 뷰 또는 저장된 프로시저를 사용 하 여 빈 데이터베이스를 수 있습니다. 이러한 기준을 결과 크게 변경 되 면 로그에 모든 데이터베이스의 테이블, 뷰 및 초기 배포 후의 변경 내용 함께 저장된 프로시저의 만들기를 포함 해야 합니다. 스펙트럼의 한쪽 끝에서 처음에 프로덕션 환경에 배포 되는 데이터베이스의 버전으로 초기 계획을 설정할 수 있습니다. 첫 번째 배포 데이터베이스에 대 한 변경만 포함 하기 때문에 훨씬 더 작은 변경 로그이 선택이 발생 합니다. 이 방법은 않겠습니다. 고 물론 선택할 수 있습니다 road 접근 방식의 자세한 중간 기준선 정의 데이터베이스의 초기 작성 및 데이터베이스 배포 되는 경우 먼저 간의 일부 지점입니다.

만든 후 선택한 초기 기준 버전을 다시 실행할 수 있는 SQL 스크립트 생성 것이 좋습니다. 이러한 스크립트의 기준 버전의 데이터베이스를 손쉽게 만들 수 있습니다. 이 기능은 특히 대규모 프로젝트에서는 유용한 데이터베이스의 자체 복사본을 필요한 각 프로젝트 또는 테스트, 스테이징 등의 추가 환경에서 작업 하는 여러 개발자가 있을 경우.

다양 한 도구 기준 버전의 SQL 스크립트를 생성 하는 데 사용할 수 있습니다. 작업 하위 메뉴에서 Management Studio SSMS (SQL Server) 데이터베이스에서 마우스 오른쪽 단추로 클릭 수, 이동한 스크립트 생성 옵션을 선택 합니다. 이 개체를 만들 데이터베이스의 SQL 명령이 포함 된 파일을 생성 하도록 지시할 수 있습니다 하는 스크립트 마법사를 시작 합니다. 뿐만 아니라 데이터베이스 테이블에서 데이터베이스 스키마 뿐만 아니라 데이터를 만드는 SQL 명령을 생성할 수 있는 데이터베이스 게시 마법사는 하는 방법도 있습니다. 데이터베이스 게시 마법사를 자세히 살펴본 년대 합니다 *데이터베이스 배포* 자습서입니다. 어떤 도구를 사용에 관계 없이 있어야 끝에서 데이터베이스의 기준 버전을 다시 사용할 수 있는 스크립트 파일은 필요에 발생 합니다.

Prose에 데이터베이스 변경 내용을 문서화의 주된 이점은 단순성입니다. 데이터베이스 개체 만들기 및 변경에 대 한 SQL 구문 사용 하 여 t 필요 경험 하지. 대신 prose에 변경 내용을 기록 하 고 SQL Server Management Studio가 그래픽 사용자 인터페이스를 통해 구현 수입니다.

## <a name="taking-the-application-offline-during-deployment"></a>동안 응용 프로그램을 오프 라인 배포를 수행합니다.

것으로이 자습서 전체에서 확인할 ve 배포는 여러 단계를 포함 하는 프로세스: 개발 환경에서 ASP.NET 페이지, 마스터 페이지, CSS 파일, JavaScript 파일, 이미지 및 기타 필요한 콘텐츠를 프로덕션에 복사 환경 필요한 경우 프로덕션 환경 특정 구성 정보를 복사 및 마지막 배포 이후의 변경 내용을 데이터 모델에 적용 합니다. 파일의 수 및 데이터베이스 변경 내용을의 복잡성에 따라이 단계 걸릴 수 있습니다 몇 초에서 몇 분 정도 완료 합니다. 이 기간 동안 웹 응용 프로그램 flux에 있으며 사이트를 방문 하는 사용자 오류 또는 예기치 않은 동작이 발생할 수 있습니다.

웹 사이트를 배포할 때 배포 완료 될 때까지 "오프 라인" 웹 응용 프로그램을 사용 하는 것이 좋습니다. 응용 프로그램을 오프 라인 (및 배포 프로세스가 완료 되 면 백업 가져오기)는 파일을 업로드 한 후 삭제 하는 것 만큼 쉽습니다. 명명 된 파일의 단순한 존재 ASP.NET 2.0 부터는 `app_offline.htm` 응용 프로그램에서 루트 디렉터리 오프 라인 전체 웹 사이트 "." 해당 사이트의 ASP.NET 페이지에 대 한 요청이 자동으로 응답의 내용으로는 `app_offline.htm` 파일입니다. 해당 파일이 제거 되 면 응용 프로그램이 다시 온라인 상태가 됩니다.
